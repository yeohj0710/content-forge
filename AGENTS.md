# AGENTS.md — Content Forge 워크플로우

> **이 문서는 AI 에이전트(Claude Code)가 이 프로젝트에서 콘텐츠를 제작할 때 반드시 따라야 하는 운영 매뉴얼이다.**
> 새 세션에서 `prompts/bootstrap-image.txt` 또는 `prompts/bootstrap-video.txt`가 붙여넣어지면,
> 에이전트는 가장 먼저 이 문서 전체를 읽고 워크플로우를 숙지한 뒤 Phase 1부터 진행한다.

---

## 0. 이 프로젝트의 목적

단일 AI 에이전트만으로도 콘텐츠 제작 전체를 수행할 수 있지만, 결과물의 품질에는 한계가 있다.
이 워크플로우는 **각 단계를 가장 잘하는 주체에게 배분**해서 품질 상한을 끌어올린다.

- **설계·조립·텍스트** → Claude Code (Figma MCP / Higgsfield MCP)
- **이미지 생성** → GPT Image 등 이미지 특화 모델 (사람이 실행)
- **에셋 수급** → 사람 (제품 누끼, 로고, 실사 컷 등)
- **영상 후편집·자막** → 사람 (편집 툴)

산출물 종류는 제한이 없다: 1:1 광고 이미지, PPT/제안서 슬라이드, 제품·광고 상세페이지,
썸네일, 배너, 숏폼 영상, 브랜드 필름 등 무엇이든 이 워크플로우로 처리한다.

## 1. 핵심 원칙 (모든 트랙 공통)

### 1-1. 텍스트-비주얼 분리 원칙 ★ 가장 중요
- **생성형 모델(GPT Image, Higgsfield 등)이 만드는 비주얼에는 텍스트가 절대 들어가면 안 된다.**
- 이미지: 텍스트는 전부 Figma의 네이티브 텍스트 레이어로 에이전트가 직접 올린다. → 언제든 편집 가능.
- 영상: 자막·타이틀은 전부 사람의 후편집으로 입힌다. 영상 모델은 글자를 제대로 렌더링하지 못한다.
- 이 원칙 덕분에 "문구 수정"이라는 가장 빈번한 수정 요청을 재생성 없이 즉시 처리할 수 있다.

### 1-2. 설계 우선 원칙
- 어떤 도구도 건드리기 전에 **설계도(blueprint.md)를 먼저 완성**한다.
- 최종 산출물의 전체 그림(캔버스 크기, 프레임 수, 레이아웃, 텍스트 위계, 스타일 방향)이
  머릿속이 아니라 **문서로** 존재해야 에셋 요구 목록이 정확해진다.

### 1-3. 에셋은 과하게 요구하는 원칙
- 에셋 매니페스트는 **풍부하고 과감하게** 작성한다. 부족한 것보다 남는 것이 낫다.
- 누끼 딴 제품컷, 로고 원본(SVG 우선), 인증마크, 질감/텍스처, 실사 컷, 브랜드 컬러·폰트 정보 등
  "있으면 퀄리티가 올라가는 것"은 전부 요구한다.

### 1-4. 프롬프트 파일 = 재현 가능성 원칙
- GPT Image로 생성할 이미지는 **이미지 1장당 프롬프트 txt 파일 1개**를 에이전트가 작성한다.
- 생성 시 함께 입력해야 하는 에셋(레퍼런스 이미지, 합성용 제품 누끼, i2v 시드 프레임 등)이 있으면
  **에이전트가 그 파일들까지 프롬프트 옆에 세팅**해둔다(§8-1 첨부 에셋 규칙).
  사람은 해당 폴더를 열고 프롬프트 + 첨부파일을 그대로 업로드만 하면 된다.
- 배경 이미지를 수정해야 하면 → 에이전트가 해당 프롬프트 txt(및 첨부 세트)를 새 버전으로 교체하고,
  사람이 다시 생성해서 파일을 갈아끼운다. 프롬프트가 파일로 남아 있어야 이 루프가 돌아간다.

### 1-5. 과적합 금지 원칙
- 과거 작업물(예: 하마하마 광고 15종 Figma 파일)은 **스타일 참고자료일 뿐 템플릿이 아니다.**
- 작업할 콘텐츠의 종류는 매번 다르다. 이전 작업의 레이아웃·구성을 복붙하지 말고
  이번 요청의 목적에 맞게 처음부터 설계한다.

### 1-6. 셀프 QA 원칙
- Figma 조립이 끝나면 사용자에게 보여주기 전에 `get_screenshot`으로 각 프레임을 캡처해서
  §10 체크리스트로 **스스로 1~2회 검수·수정**한 뒤에 공유한다.

## 3. 저장소 구조와 작업 폴더 규칙

```
content-forge/
├── AGENTS.md                  ← 이 문서
├── README.md
├── prompts/
│   ├── bootstrap-image.txt    ← 이미지 트랙 시작용 (새 세션에 붙여넣기)
│   └── bootstrap-video.txt    ← 영상 트랙 시작용
├── templates/                 ← 산출 문서 템플릿 (에이전트가 참고)
└── projects/                  ← 작업 단위 폴더 (git 미추적)
    └── YYYYMMDD-작업명/
        ├── brief.md           ← 사용자 요청 원문 + 확정 사양
        ├── blueprint.md       ← 설계도
        ├── asset-manifest.md  ← 에셋 체크리스트 (상태 추적)
        ├── assets/
        │   ├── provided/      ← 사람이 수급한 원본 에셋
        │   ├── generated/     ← GPT Image 생성 결과물
        │   └── prompts/       ← 생성 이미지별 프롬프트 txt (1:1 대응)
        ├── edit-guide.md      ← (영상 트랙) 편집 가이드
        └── output/            ← 최종 export
```

- 새 작업이 시작되면 에이전트가 `projects/YYYYMMDD-작업명/` 폴더와 하위 구조를 즉시 생성한다.
- 에셋 파일명은 매니페스트의 ID와 반드시 일치시킨다. (예: `P01-product-cutout.png`, `G03-hero-bg-v2.png`)
- 재생성된 파일은 덮어쓰지 말고 `-v2`, `-v3` 접미사로 버전을 남긴다. 프롬프트 txt도 동일하게 버전링한다.

## 4. 공통 파이프라인 개요

```
[Phase 0] 부트스트랩          사람: bootstrap txt + 요청 프롬프트 붙여넣기
     ↓
[Phase 1] 설계               AI: AGENTS.md 숙지 → 질문(필요시) → blueprint.md
     ↓
[Phase 2] 에셋 매니페스트      AI: asset-manifest.md + 생성이미지 프롬프트 txt들
     ↓
[Phase 3] 에셋 준비           사람: 수급 + GPT Image 생성 → assets/에 배치
     ↓
[Phase 4] 에셋 검수           AI: 매니페스트 대조, 누락·품질 문제 보고
     ↓
[Phase 5] 제작               AI: Figma 조립 (이미지) / Higgsfield 클립 생성 (영상)
     ↓
[Phase 6] 셀프 QA → 공유      AI: 스크린샷 검수 → 사용자 리뷰 / (영상) edit-guide.md 전달
     ↓
[Phase 7] 수정 루프           텍스트: AI 즉시 수정 ｜ 비주얼: 프롬프트 교체 → 재생성 → 교체
```

## 5. 이미지 트랙 (Figma MCP) — Phase 상세

### Phase 1 — 설계
1. 사용자 요청에서 다음을 확정한다. 불명확하면 **작업 시작 전에** 질문한다:
   - 산출물 종류(광고/슬라이드/상세페이지 등), 장수, 크기·비율(예: 1080×1080, 16:9)
   - 타겟·톤앤매너, 브랜드 제약(컬러·폰트·로고 사용 규정)
   - 확정 카피가 있는지, 에이전트가 카피라이팅까지 하는지
2. `blueprint.md` 작성 (templates/design-blueprint.md 참고):
   - 프레임별 레이아웃 스케치(텍스트 위치, 비주얼 위치, 여백 계획)
   - 디자인 토큰: 컬러 팔레트(hex), 폰트 페어링, 텍스트 위계(H1/H2/Body/Caption 크기)
   - 각 프레임에서 배경 이미지 위에 텍스트가 올라갈 영역 → 이 정보가 Phase 2 프롬프트의 "여백 지시"가 된다

### Phase 2 — 에셋 매니페스트
1. `asset-manifest.md` 작성. 두 종류로 구분:
   - **P(provided) 에셋**: 사람이 수급. 제품 누끼(투명 PNG, 최소 1500px), 로고(SVG 우선),
     인증마크, 실사 컷, 아이콘, 질감 소스 등. 요구 스펙(형식·해상도·배경투명 여부)을 항목마다 명시.
   - **G(generated) 에셋**: GPT Image로 생성. 각 항목마다 `assets/prompts/G##-이름.txt` 프롬프트 파일을
     에이전트가 **이 단계에서 전부 작성**한다. (작성 규칙은 §8)
     함께 입력할 첨부 에셋(레퍼런스·합성용 누끼 등)이 있으면 §8-1 규칙대로 첨부 폴더까지 세팅한다.
2. 매니페스트에는 체크박스를 두어 준비 상태를 추적한다.
3. 사용자에게 매니페스트를 보여주고 "이 목록대로 준비해달라"고 명확히 요청한 뒤 **대기**한다.

### Phase 3 — 에셋 준비 (사람)
- 사람이 P 에셋을 `assets/provided/`에, GPT Image 생성 결과를 `assets/generated/`에 저장한다.
- 파일명은 매니페스트 ID를 따른다.

### Phase 4 — 에셋 검수
- 폴더를 스캔해서 매니페스트와 대조. 누락·파일명 불일치·해상도 부족·배경 미투명 등을 표로 보고.
- 문제가 있으면 해결될 때까지 Phase 5로 넘어가지 않는다.

### Phase 5 — Figma 조립
1. **`use_figma` 호출 전 반드시 `/figma-use` 스킬을, 새 파일 생성 전 `/figma-create-new-file` 스킬을 로드한다.**
2. 파일 생성 규칙: 파일명은 생성 시점에만 지정 가능하므로, 생성 시 이름에 **`[Claude]` 마커를 포함**한다.
   (다른 AI 도구와 동시 작업 시 구분용. 예: `[Claude] 리쥬닉 상세페이지 v1`)
3. 조립 순서:
   - 프레임 생성 (blueprint의 크기·장수대로)
   - `upload_assets`로 에셋 업로드 → 배경은 이미지 필, 누끼는 개별 레이어로 배치
   - **모든 텍스트는 네이티브 텍스트 레이어로 생성** (이미지에 구운 텍스트 금지)
   - 텍스트 스타일을 일관되게: blueprint의 위계(H1/H2/Body)를 그대로 적용
   - 반복 요소(카드, 리스트 아이템)는 오토레이아웃 사용 → 문구 길이가 바뀌어도 안 깨지게
4. 한글 타이포그래피: 어절 단위 줄바꿈, 자간 살짝 좁게(-2~-4%), 본문 행간 150~170%.

### Phase 6 — 셀프 QA → 공유
- `get_screenshot`으로 전 프레임 캡처 → §10 체크리스트로 자체 검수 → 수정 → 재캡처.
- 통과하면 Figma 링크와 함께 사용자에게 공유. 프레임별로 의도(왜 이렇게 배치했는지)를 한 줄씩 요약.

### Phase 7 — 수정 루프
- **문구·색·배치 수정**: Figma에서 즉시 수정. 재생성 불필요.
- **배경 이미지 수정**: 해당 `G##` 프롬프트 txt를 새 버전(`-v2`)으로 작성 → 사람이 재생성 →
  `assets/generated/`에 v2로 저장 → 에이전트가 Figma에서 이미지 필만 교체.
- 수정이 끝날 때마다 매니페스트의 버전 기록을 갱신한다.

## 6. 영상 트랙 (Higgsfield MCP) — Phase 상세

> 이미지 트랙과 Phase 0~4는 동일하다. 차이점만 기술한다.
> **영상 트랙에서 에이전트는 "설계자"다. 최종 편집은 사람이 한다.**

### Phase 1 — 설계 (스토리보드)
`blueprint.md`에 다음을 포함:
- 영상 목적·타겟·총 길이·비율(9:16 / 16:9 / 1:1)·플랫폼(릴스, 유튜브 등)
- **씬 리스트**: 씬 번호, 길이(초), 화면 내용, 카메라 무브, 이 씬의 역할(훅/전개/CTA)
- 씬별 생성 방식: t2v(텍스트→영상) vs i2v(이미지→영상 — 제품컷 등 시드 이미지 필요)
- i2v용 시드 이미지는 P 에셋으로 매니페스트에 올린다

### Phase 2~4 — 에셋
- i2v 시드용 제품컷·레퍼런스 이미지가 핵심 P 에셋.
- 필요 시 GPT Image로 씬 배경·시드 이미지를 먼저 뽑는 하이브리드도 가능 (이미지 트랙 §8 규칙 동일).

### Phase 5 — 클립 생성
1. Higgsfield MCP로 씬별 클립을 생성한다. 1개일 수도, 여러 개를 이어붙일 수도 있다.
2. **클립 프롬프트에 텍스트·글자·간판·라벨이 나오게 하는 요소를 넣지 않는다.**
   네거티브로 `no text, no letters, no captions, no watermarks`를 명시한다.
3. 각 클립의 생성 프롬프트도 `assets/prompts/V##-이름.txt`로 남긴다 (재생성 대비).
   i2v 시드 프레임 등 함께 입력할 파일이 있으면 §8-1 규칙대로 `V##-이름-attachments/` 폴더에
   세팅한다. (MCP로 에이전트가 직접 생성하는 경우에도 동일하게 남긴다 — 사람이 외부 툴에서
   재생성해야 할 때 그대로 쓸 수 있는 세트가 되도록.)
4. 결과 클립은 `output/clips/`에 씬 번호 순으로 저장·정리한다.

### Phase 6 — 편집 가이드 산출 ★ 이 트랙의 최종 산출물
`edit-guide.md` 작성 (templates/edit-guide.md 참고). 반드시 포함:
1. **타임라인 표**: 순서 | 클립 파일명 | 사용 구간(트림 in/out) | 길이
2. **자막 대본**: 타임코드별 자막 텍스트 + 위치(상/중/하) + 스타일 제안(크기·색·강조어)
3. **트랜지션**: 클립 사이 전환 방식 (컷/디졸브/와이프 등)
4. **사운드**: BGM 분위기·전환 포인트, 효과음 위치
5. **내보내기 설정**: 해상도, fps, 코덱, 플랫폼별 스펙
- 사람이 이 가이드만 보고 편집 툴에서 그대로 작업할 수 있을 만큼 구체적으로 쓴다.

### Phase 7 — 수정 루프
- **자막·편집 수정**: 사람이 편집 툴에서 직접 (가이드 갱신이 필요하면 에이전트가 edit-guide.md 수정).
- **클립 자체 수정**: 해당 `V##` 프롬프트를 새 버전으로 교체 → 재생성 → 가이드의 파일명 갱신.

## 7. 에셋 매니페스트 작성 규칙

- ID 체계: `P##`(사람 수급) / `G##`(GPT Image 생성) / `V##`(영상 클립)
- 항목마다: ID, 파일명, 용도(어느 프레임/씬), 요구 스펙, 준비 상태 체크박스
- **풍부하게 요구할 것.** 판단 기준: "이게 있으면 결과물이 더 있어 보이는가?" → 그렇다면 목록에 넣는다.
  - 제품이 있다면: 정면 누끼 + 사선 누끼 + 디테일 컷까지
  - 브랜드가 있다면: 로고 가로형/세로형/심볼, 브랜드 컬러 hex, 지정 폰트
  - 분위기가 필요하면: 질감(종이/패브릭/그라데이션), 보조 오브젝트 누끼
- 스펙 기본값: 이미지 최소 1500px(장변), 누끼는 투명 PNG, 로고는 SVG 우선, 배경은 사용 크기의 2배 해상도

## 8. GPT Image 프롬프트 txt 작성 규칙

파일 위치·이름: `assets/prompts/G##-용도.txt` (이미지 1장 = txt 1개)

각 txt의 고정 포맷:

```
[용도] 어느 프레임의 무엇인지 (예: 프레임 3 히어로 배경)
[크기/비율] 예: 2048x2048, 1:1
[스타일] 스타일 키워드, 조명, 무드, 컬러 톤 (blueprint의 디자인 토큰과 일치시킬 것)
[구도/여백] 텍스트가 올라갈 영역을 비워두라는 지시 (예: "상단 1/3은 단순한 하늘로 비워둘 것")
[첨부 에셋] 함께 업로드할 파일 목록 + 각 파일의 역할 (없으면 "없음")
  예: attachments/ref-01-product.png — 이 제품을 이미지 중앙에 자연스럽게 합성할 것
      attachments/ref-02-mood.jpg — 색감·조명 스타일 레퍼런스로만 참고할 것
[프롬프트]
(여기에 실제 생성 프롬프트 전문 — 영어 권장. 첨부 에셋이 있으면 프롬프트 본문에서도
 각 첨부의 역할을 명시할 것: "use the attached product photo as the subject...")
[네거티브/금지]
no text, no letters, no typography, no logos, no watermarks, no captions
```

원칙:
- **배경/비주얼 전용. 텍스트가 렌더링될 여지를 프롬프트 차원에서 차단한다.**
- 텍스트를 올릴 영역의 여백(네거티브 스페이스)을 반드시 구도 지시에 포함한다.
- 같은 작업 내 이미지들은 스타일 키워드를 공유해서 톤이 통일되게 한다.
- 수정 시: 기존 txt를 지우지 말고 `G##-용도-v2.txt`로 새로 작성한다. 첨부 세트가 바뀌면 첨부 폴더도 v2로.

### 8-1. 첨부 에셋 세팅 규칙 (이미지 G## / 영상 V## 공통)

생성 모델에 프롬프트와 **함께 입력해야 하는 파일**이 있는 경우(image-to-image 합성, 스타일
레퍼런스, i2v 시드 프레임 등), 에이전트가 프롬프트 작성 시점에 다음까지 세팅한다:

```
assets/prompts/
├── G03-hero-bg.txt                  ← 프롬프트 (첨부 없으면 이 파일 하나로 끝)
└── G03-hero-bg-attachments/         ← 첨부가 있으면 에이전트가 이 폴더를 만들고 파일을 복사해둔다
    ├── ref-01-product.png           ← assets/provided/의 P 에셋에서 복사 (원본은 그대로 둠)
    └── ref-02-mood.jpg
```

- 첨부 후보: P 에셋(제품 누끼·실사 컷), 기존 G 에셋(앞서 생성한 이미지를 시드로 재사용),
  외부 레퍼런스 이미지. **원본을 옮기지 말고 복사**해서 세팅한다 (원본 위치는 매니페스트 기준 유지).
- 파일명은 `ref-##-역할` 형식으로 다시 지어, txt의 [첨부 에셋] 목록과 1:1로 대응시킨다.
- 사람의 실행 절차가 "폴더 열기 → 프롬프트 복사 → 첨부 전부 업로드 → 생성"으로 끝나야 한다.
  첨부의 용도 구분(합성 대상인지, 스타일 참고인지)은 txt에 이미 적혀 있으므로 사람이 고민할 게 없어야 한다.
- 아직 준비되지 않은 P 에셋을 첨부해야 하는 경우: 매니페스트에 의존 관계를 명시하고
  ("G03은 P01 수급 후 첨부 세팅 예정"), P 에셋 도착 시 에이전트가 첨부 폴더를 완성한다.

## 9. Figma 작업 규칙 요약

1. `/figma-use` 스킬 로드 없이 `use_figma`를 호출하지 않는다. 새 파일은 `/figma-create-new-file` 먼저.
2. 파일명은 생성 시에만 지정 가능 → 생성 시 `[Claude]` 마커 포함.
3. 텍스트는 100% 네이티브 텍스트 레이어. 이미지에 텍스트를 굽지 않는다.
4. 레이어 이름을 의미 있게 짓는다 (`H1-메인카피`, `BG-hero`, `IMG-P01-product`).
5. 반복 구조는 오토레이아웃. 색·폰트는 blueprint 토큰만 사용 (임의 색 추가 금지).
6. 프레임 이름은 순번+역할 (`01-훅`, `02-문제제기`, ...).

## 10. 셀프 QA 체크리스트 (공유 전 필수)

- [ ] 텍스트 오버플로/잘림 없음, 어절 단위 줄바꿈

- [ ] 누끼 가장자리 깨짐·저해상도 업스케일 흔적 없음

- [ ] 로고·인증마크 등 필수 요소 누락 없음
- [ ] (영상) 클립에 글자·워터마크 없음, 씬 연결 시 톤 점프 없음
