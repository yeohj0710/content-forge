# AGENTS.md — Content Forge 워크플로우

> **이 문서는 AI 에이전트(Claude Code)가 이 프로젝트에서 콘텐츠를 제작할 때 반드시 따라야 하는 운영 매뉴얼이다.**
> 새 세션에서 `prompts/bootstrap-image.txt` 또는 `prompts/bootstrap-video.txt`가 붙여넣어지면,
> 에이전트는 가장 먼저 이 문서 전체를 읽고, **산출물 종류에 해당하는 `playbooks/` 문서와
> `standards/design-standards.md`를 읽은 뒤** Phase 1부터 진행한다.

---

## 0. 이 프로젝트의 목적

단일 AI 에이전트만으로도 콘텐츠 제작 전체를 수행할 수 있지만, 결과물의 품질에는 한계가 있다.
이 워크플로우는 **각 단계를 가장 잘하는 주체에게 배분**해서 품질 상한을 끌어올린다.

- **리서치·설계·조립·텍스트** → Claude Code (Figma MCP / Higgsfield MCP)
- **이미지 생성** → GPT Image 등 이미지 특화 모델 (사람이 실행)
- **에셋 수급** → 사람 (제품 누끼, 로고, 실사 컷, 인증서 스캔 등)
- **영상 후편집·자막** → 사람 (편집 툴)

산출물 종류는 제한이 없다: 1:1 광고 이미지, PPT/제안서 슬라이드, 제품·광고 상세페이지,
썸네일, 배너, 숏폼 영상, 브랜드 필름 등 무엇이든 이 워크플로우로 처리한다.

**품질 게이트**: 통과 기준은 언제나 "이대로 실제 스토어/미팅/광고에 올릴 수 있는가"다.
"AI치고 잘 만들었다"는 기준이 아니다. (standards/design-standards.md §1)

## 1. 문서 지도 — 무엇을 언제 읽는가

| 문서 | 언제 |
|---|---|
| `AGENTS.md` (이 문서) | 모든 작업 시작 시 전체 숙지 |
| `standards/design-standards.md` | 모든 작업 시작 시 — 공통 품질 기준(타이포·밀도·대비·QA 절차) |
| `playbooks/detail-page.md` | 제품 상세페이지 작업 시 |
| `playbooks/deck.md` | PPT·협업/마케팅 제안서·IR 등 슬라이드 작업 시 |
| `playbooks/ad-image.md` | 단장 광고·배너·썸네일 작업 시 |
| `templates/` | 각 Phase 산출 문서 작성 시 참고 |

플레이북이 없는 산출물 종류(예: 새로운 형식)는 가장 가까운 플레이북을 참고하되,
작업 후 배운 것을 새 플레이북으로 남길지 사용자에게 제안한다.

## 2. 핵심 원칙 (모든 트랙 공통)

### 2-1. 텍스트-비주얼 분리 원칙 ★ 가장 중요
- **생성형 모델(GPT Image, Higgsfield 등)이 만드는 비주얼에는 텍스트가 절대 들어가면 안 된다.**
- 이미지: 텍스트는 전부 Figma의 네이티브 텍스트 레이어로 에이전트가 직접 올린다. → 언제든 편집 가능.
- 영상: 자막·타이틀은 전부 사람의 후편집으로 입힌다. 영상 모델은 글자를 제대로 렌더링하지 못한다.
- 이 원칙 덕분에 "문구 수정"이라는 가장 빈번한 수정 요청을 재생성 없이 즉시 처리할 수 있다.

### 2-2. 리서치 우선 원칙 ★ 카피는 팩트에서 나온다
- **셀링포인트 없는 설계는 시작하지 않는다.** 제품 팩트(수치·인증·원료·스토리·증거)를 수집해
  research.md를 완성하기 전에는 blueprint를 쓰지 않는다.
- 팩트가 부족하면 사용자에게 요구한다. **지어내지 않는다** — 근거 없는 카피("프리미엄", "많은 분들이
  선택")는 지면 낭비다. 강한 페이지는 "300,000ppm", "경쟁사 평균의 20배", "리뷰 1,200개"로 말한다.

### 2-3. 설계 우선 원칙
- 어떤 도구도 건드리기 전에 **설계도(blueprint.md)를 먼저 완성**한다.
- 최종 산출물의 전체 그림(캔버스 크기, 프레임 수, 레이아웃, 텍스트 위계, 스타일 방향)이
  머릿속이 아니라 **문서로** 존재해야 에셋 요구 목록이 정확해진다.

### 2-4. 에셋은 과하게 요구하는 원칙
- 에셋 매니페스트는 **풍부하고 과감하게** 작성한다. 부족한 것보다 남는 것이 낫다.
- 누끼·연출컷·로고 원본(SVG)·**인증서 실물 스캔·논문 캡처·리뷰 캡처** 등 신뢰·증거 에셋까지
  "있으면 퀄리티가 올라가는 것"은 전부 요구한다. 증거는 텍스트보다 실물 이미지가 힘이 세다.
- **생성(G) 에셋도 아끼지 않는다 — GPT Image 생성은 사실상 무제한이다.** 생성 비용을 이유로
  G 에셋 수를 줄이지 말 것. 비율 상한(2:3) 안에 들어오는 지면은 **풀블리드 생성 배경을 기본
  후보**로 검토하고(단색 배경은 의도적 선택일 때만), 프레임·섹션마다 배경/오브젝트/질감을
  풍부하게 계획한다. 같은 G 에셋도 베스트 픽을 위해 여러 번 생성(-v2, -v3)을 권장한다.
  텍스트가 직접 올라가는 영역의 여백 지시(§9)만 지키면 리치한 배경이 곧 퀄리티다.

### 2-5. 프롬프트 파일 = 재현 가능성 원칙
- GPT Image로 생성할 이미지는 **이미지 1장당 프롬프트 txt 파일 1개**를 에이전트가 작성한다.
- 생성 시 함께 입력해야 하는 에셋(레퍼런스 이미지, 합성용 제품 누끼, i2v 시드 프레임 등)이 있으면
  **에이전트가 그 파일들까지 프롬프트 옆에 세팅**해둔다(§9-1 첨부 에셋 규칙).
  사람은 해당 폴더를 열고 프롬프트 + 첨부파일을 그대로 업로드만 하면 된다.
- 배경 이미지를 수정해야 하면 → 에이전트가 해당 프롬프트 txt(및 첨부 세트)를 새 버전으로 교체하고,
  사람이 다시 생성해서 파일을 갈아끼운다. 프롬프트가 파일로 남아 있어야 이 루프가 돌아간다.

### 2-6. 레퍼런스 벤치마킹 원칙 (과적합 금지와 세트)
- 작업 시작 시 레퍼런스 2~3개를 요구하고, 타이포 비율·밀도·증거 표현을 **측정**해서
  기준선(bar)으로 삼는다(standards §6). 목표는 기준선을 넘는 것.
- 단, **레퍼런스와 과거 작업물은 기준선 측정용이지 템플릿이 아니다.** 레이아웃·구성을 복붙하지
  말고 이번 요청의 목적에 맞게 처음부터 설계한다.

### 2-7. 셀프 QA 원칙
- Figma 조립이 끝나면 사용자에게 보여주기 전에 `get_screenshot`으로 각 프레임을 캡처해서
  §11 체크리스트로 **스스로 1~2회 검수·수정**한 뒤에 공유한다.
- **축소 판독 테스트 필수**: maxDimension 400(광고류는 200)으로 다시 캡처해 모바일/썸네일
  크기에서 판독되는지 확인한다. 원본에서만 읽히는 텍스트는 실패다. (standards §7)

## 4. 저장소 구조와 작업 폴더 규칙

```
content-forge/
├── AGENTS.md                  ← 이 문서
├── README.md
├── standards/
│   └── design-standards.md    ← 공통 디자인 표준 (타이포·밀도·QA)
├── playbooks/                 ← 산출물 종류별 전용 기준
│   ├── detail-page.md         ← 제품 상세페이지
│   ├── deck.md                ← PPT·제안서
│   └── ad-image.md            ← 광고·배너·썸네일
├── prompts/
│   ├── bootstrap-image.txt    ← 이미지 트랙 시작용 (새 세션에 붙여넣기)
│   └── bootstrap-video.txt    ← 영상 트랙 시작용
├── templates/                 ← 산출 문서 템플릿 (에이전트가 참고)
└── projects/                  ← 작업 단위 폴더 (git 미추적)
    └── YYYYMMDD-작업명/
        ├── brief.md           ← 사용자 요청 원문 + 확정 사양
        ├── research.md        ← 리서치 & 카피 전략 ★ blueprint의 선행 조건
        ├── blueprint.md       ← 설계도
        ├── asset-manifest.md  ← 에셋 체크리스트 (상태 추적)
        ├── references/        ← 벤치마킹 레퍼런스 (사람이 수급, 에이전트가 분석)
        ├── assets/
        │   ├── provided/      ← 사람이 수급한 원본 에셋
        │   ├── generated/     ← GPT Image 생성 결과물
        │   └── prompts/       ← 생성 이미지별 프롬프트 txt (1:1 대응)
        ├── edit-guide.md      ← (영상 트랙) 편집 가이드
        └── output/            ← 최종 export
```

- 새 작업이 시작되면 에이전트가 `projects/YYYYMMDD-작업명/` 폴더와 하위 구조를 즉시 생성한다.
- 에셋 파일명은 매니페스트의 ID와 반드시 일치시킨다. (예: `P01-product-cutout.png`, `G03-hero-bg-v2.png`)
- 재생성된 파일은 덮어쓰지 말고 `-v2`, `-v3` 접미사로 버전을 남긴다. 프롬프트 txt도 동일하게 버전링한다.

## 5. 공통 파이프라인 개요

```
[Phase 0] 부트스트랩          사람: bootstrap txt + 요청 프롬프트 붙여넣기
     ↓
[Phase 1] 리서치·카피 전략    AI: AGENTS.md+표준+플레이북 숙지 → 팩트·레퍼런스 수집(질문)
                                → research.md (셀링포인트 랭킹 + 확정 카피)
     ↓
[Phase 2] 설계               AI: blueprint.md (플레이북 아키텍처 + 타이포 스케일 적용)
     ↓
[Phase 3] 에셋 매니페스트      AI: asset-manifest.md + 생성이미지 프롬프트 txt들
     ↓
[Phase 4] 에셋 준비           사람: 수급 + GPT Image 생성 → assets/에 배치
     ↓
[Phase 5] 에셋 검수           AI: 매니페스트 대조, 누락·품질 문제 보고
     ↓
[Phase 6] 제작               AI: Figma 조립 (이미지) / Higgsfield 클립 생성 (영상)
     ↓
[Phase 7] 셀프 QA → 공유      AI: 스크린샷+축소 판독 검수 → 사용자 리뷰 / (영상) edit-guide.md
     ↓
[Phase 8] 수정 루프           텍스트: AI 즉시 수정 ｜ 비주얼: 프롬프트 교체 → 재생성 → 교체
```

Phase 1~3은 사용자와의 핑퐁이 허용된다(팩트 질문, 카피 방향 확인). Phase 4 이후는
매니페스트가 확정된 상태여야 하며, 에셋 검수 통과 전에 Phase 6을 시작하지 않는다.

## 6. 이미지 트랙 (Figma MCP) — Phase 상세

### Phase 1 — 리서치·카피 전략 ★ 새 작업의 첫 산출물
1. 산출물 종류를 확정하고 해당 **플레이북과 standards를 읽는다.**
2. `research.md` 작성 (templates/research.md):
   - 제품 팩트 시트: 수치·원료·인증·스토리·가격·사회적 증거. **부족하면 사용자에게 질문**
     (제품소개서·인증서·기존 상세페이지 등 자료 요청). 지어내지 않는다.
   - 타겟·페인포인트·반론, 경쟁/대안 비교(숫자로).
   - 레퍼런스 요구·분석: references/에 받은 레퍼런스의 타이포 비율·밀도·구성을 측정 기록.
   - **셀링포인트 랭킹**: [주장+숫자+근거 에셋] 세트만 등재. 근거 없으면 제외.
   - 설득 플로우와 섹션별 **확정 카피** 초안까지.
3. 사용자 확인을 받는다: 셀링포인트 순위와 카피 방향이 맞는지. 확인 후 Phase 2로.

### Phase 2 — 설계
1. 사용자 요청·research.md에서 다음을 확정한다. 불명확하면 **작업 시작 전에** 질문한다:
   - 산출물 종류(광고/슬라이드/상세페이지 등), 장수, 크기·비율 — 플레이북 규격 기본값 사용
   - 타겟·톤앤매너, 브랜드 제약(컬러·폰트·로고 사용 규정)
2. `blueprint.md` 작성 (templates/design-blueprint.md 참고):
   - 프레임별 레이아웃 스케치(텍스트 위치, 비주얼 위치, 여백 계획) — 플레이북의 아키텍처를 따름
   - 디자인 토큰: 컬러 팔레트(hex), 폰트 페어링, **타이포 스케일(플레이북 표 그대로, px 명기)**
   - 각 프레임에서 배경 이미지 위에 텍스트가 올라갈 영역 → 이 정보가 Phase 3 프롬프트의 "여백 지시"가 된다

### Phase 3 — 에셋 매니페스트
1. `asset-manifest.md` 작성. 두 종류로 구분:
   - **P(provided) 에셋**: 사람이 수급. 제품 누끼·연출컷(투명 PNG, 최소 1500px), 로고(SVG 우선),
     **인증서 스캔·논문 캡처·리뷰 캡처 등 증거 에셋**, 실사 컷, 아이콘, 질감 소스 등.
     요구 스펙(형식·해상도·배경투명 여부)을 항목마다 명시.
   - **G(generated) 에셋**: GPT Image로 생성. 각 항목마다 `assets/prompts/G##-이름.txt` 프롬프트 파일을
     에이전트가 **이 단계에서 전부 작성**한다. (작성 규칙은 §9)
     함께 입력할 첨부 에셋(레퍼런스·합성용 누끼 등)이 있으면 §9-1 규칙대로 첨부 폴더까지 세팅한다.
2. 매니페스트에는 체크박스를 두어 준비 상태를 추적한다.
3. 사용자에게 매니페스트를 보여주고 "이 목록대로 준비해달라"고 명확히 요청한 뒤 **대기**한다.

### Phase 4 — 에셋 준비 (사람)
- 사람이 P 에셋을 `assets/provided/`에, GPT Image 생성 결과를 `assets/generated/`에 저장한다.
- 파일명은 매니페스트 ID를 따른다.

### Phase 5 — 에셋 검수
- 폴더를 스캔해서 매니페스트와 대조. 누락·파일명 불일치·해상도 부족·배경 미투명 등을 표로 보고.
- 문제가 있으면 해결될 때까지 Phase 6으로 넘어가지 않는다.

### Phase 6 — Figma 조립
1. **`use_figma` 호출 전 반드시 `/figma-use` 스킬을, 새 파일 생성 전 `/figma-create-new-file` 스킬을 로드한다.**
2. 파일 생성 규칙: 파일명은 생성 시점에만 지정 가능하므로, 생성 시 이름에 **`[Claude]` 마커를 포함**한다.
   (다른 AI 도구와 동시 작업 시 구분용. 예: `[Claude] 리쥬닉 상세페이지 v1`)
3. 조립 순서:
   - 프레임 생성 (blueprint의 크기·장수대로)
   - `upload_assets`로 에셋 업로드 → 배경은 이미지 필, 누끼는 개별 레이어로 배치
   - **모든 텍스트는 네이티브 텍스트 레이어로 생성** (이미지에 구운 텍스트 금지)
   - 텍스트 스타일을 일관되게: blueprint의 타이포 스케일을 px 그대로 적용.
     **조립 중 임의로 축소 금지** — 공간이 부족하면 카피를 줄이거나 레이아웃을 바꾼다.
   - 반복 요소(카드, 리스트 아이템)는 오토레이아웃 사용 → 문구 길이가 바뀌어도 안 깨지게.
     카드·표는 내용 높이에 맞춰 hug — 바닥 빈 공간 금지.
4. 한글 타이포그래피: 어절 단위 줄바꿈, 자간 살짝 좁게(-2~-4%), 본문 행간 150~170%.

### Phase 7 — 셀프 QA → 공유
- `get_screenshot`으로 전 프레임 캡처 → §11 체크리스트로 자체 검수 → 수정 → 재캡처.
- **축소 판독 테스트**(maxDimension 400, 광고류 200)로 모바일 판독성 확인.
- 레퍼런스와 나란히 비교 — 밀도·타이포·증거에서 밀리면 리워크 후 공유.
- 통과하면 Figma 링크와 함께 사용자에게 공유. 프레임별로 의도(왜 이렇게 배치했는지)를 한 줄씩 요약.

### Phase 8 — 수정 루프
- **문구·색·배치 수정**: Figma에서 즉시 수정. 재생성 불필요.
- **배경 이미지 수정**: 해당 `G##` 프롬프트 txt를 새 버전(`-v2`)으로 작성 → 사람이 재생성 →
  `assets/generated/`에 v2로 저장 → 에이전트가 Figma에서 이미지 필만 교체.
- 수정이 끝날 때마다 매니페스트의 버전 기록을 갱신한다.

### 6-1. 긴 캔버스 특칙 — 상세페이지·롱폼 콘텐츠 ★

> 상세 규격·섹션 아키텍처·타이포 스케일은 `playbooks/detail-page.md`가 기준이다.
> 여기서는 생성 이미지 비율 한계 대응 원칙만 요약한다.

상세페이지는 세로 수천~수만 px의 긴 이미지지만, **GPT Image는 장변 ~1536px(약 2:3)이 상한**이다.
긴 이미지를 한 장으로 뽑으려 하지 말 것 — 프롬프트로 요구해도 안 되고, 아웃페인팅으로 이어붙이는
방식은 톤이 드리프트해서 품질이 무너진다. 대신:

1. **섹션 스택 설계**: 페이지를 섹션 단위(폭 1000px, 섹션당 세로 1000~1600px)로 쪼개고
   Figma 세로 오토레이아웃으로 쌓는다. 섹션 = 수정·재생성·export의 기본 단위.
2. **배경 기본값은 Figma 네이티브**(단색·그라데이션·벡터). 생성 이미지는 "사진적 리치니스"가
   필요한 지점의 비주얼 오브젝트이거나, **2:3 이하로 설계된 실사 섹션의 풀블리드**로만 쓴다
   (섹션 1000×1500 ↔ 생성 1024×1536 1:1 매칭). §2-4의 "풀블리드 생성 기본값"은 **섹션 단위**로
   적용한다 — 2:3 이내 섹션이라면 풀블리드 생성 배경을 아끼지 말되, 페이지 전체의 연속성은
   여전히 Figma 네이티브 배경 + 토큰이 담당한다.
3. **이음새(seam) 처리**: 생성 이미지를 섹션 배경으로 쓸 때는 프롬프트에 반드시
   "상단/하단이 단색 #hex로 부드럽게 페이드아웃"을 지시한다(§9). 그 단색을 이웃 섹션의
   Figma 네이티브 배경색과 일치시키면 경계가 보이지 않는다.
4. **톤 연속성**: 페이지가 하나로 보이게 하는 것은 긴 배경이 아니라 디자인 토큰이다.
5. **분할 export 대비**: 섹션 경계 = 분할 경계로 설계한다.

## 8. 에셋 매니페스트 작성 규칙

- ID 체계: `P##`(사람 수급) / `G##`(GPT Image 생성) / `V##`(영상 클립)
- 항목마다: ID, 파일명, 용도(어느 프레임/씬), 요구 스펙, 준비 상태 체크박스
- **풍부하게 요구할 것.** 판단 기준: "이게 있으면 결과물이 더 있어 보이는가?" → 그렇다면 목록에 넣는다.
  - 제품이 있다면: 정면 누끼 + 사선 누끼 + 디테일 컷 + **스튜디오 연출컷**(제품+원물+액체 등)
  - 브랜드가 있다면: 로고 가로형/세로형/심볼, 브랜드 컬러 hex, 지정 폰트
  - **신뢰·증거가 필요하면(상세페이지·제안서는 항상): 인증서 실물 스캔, 특허증, 논문·시험성적서
    캡처, 리뷰·판매량 캡처, 입점처 로고, Before/After 컷** — 실물 이미지가 텍스트보다 강하다
  - 분위기가 필요하면: 질감(종이/패브릭/그라데이션), 보조 오브젝트 누끼
- 스펙 기본값: 이미지 최소 1500px(장변), 누끼는 투명 PNG, 로고는 SVG 우선, 배경은 사용 크기의 2배 해상도

## 9. GPT Image 프롬프트 txt 작성 규칙

파일 위치·이름: `assets/prompts/G##-용도.txt` (이미지 1장 = txt 1개)

각 txt의 고정 포맷:

```
[용도] 어느 프레임의 무엇인지 (예: 프레임 3 히어로 배경)
[크기/비율] 예: 2048x2048, 1:1
[스타일] 스타일 키워드, 조명, 무드, 컬러 톤 (blueprint의 디자인 토큰과 일치시킬 것)
[구도/여백] 텍스트가 올라갈 영역을 비워두라는 지시 (예: "상단 1/3은 단순한 하늘로 비워둘 것")
[첨부 에셋] 함께 업로드할 파일 목록 + 각 파일의 역할 (없으면 "없음")
  예: attachments/ref-01-product.png — 이 제품을 이미지 중앙에 자연스럽게 합성할 것
      attachments/ref-02-mood.jpg — 색감·조명 스타일 레퍼런스로만 참고할 것
[프롬프트]
(여기에 실제 생성 프롬프트 전문 — 영어 권장. 첨부 에셋이 있으면 프롬프트 본문에서도
 각 첨부의 역할을 명시할 것: "use the attached product photo as the subject...")
[네거티브/금지]
no text, no letters, no typography, no logos, no watermarks, no captions
```

원칙:
- **배경/비주얼 전용. 텍스트가 렌더링될 여지를 프롬프트 차원에서 차단한다.**
- 텍스트를 올릴 영역의 여백(네거티브 스페이스)을 반드시 구도 지시에 포함한다.
- 같은 작업 내 이미지들은 스타일 키워드를 공유해서 톤이 통일되게 한다.
- **크기 한계**: GPT Image는 장변 ~1536px, 세로는 2:3 비율이 상한이다. 이보다 긴 이미지를
  요구하는 프롬프트를 쓰지 말 것. 긴 캔버스는 §6-1 + 플레이북의 배경 전략으로 해결한다.
- 섹션 배경용 이미지는 [구도/여백]에 이음새 지시를 포함한다:
  (예: "하단 20%는 단색 #F7F4EF로 부드럽게 페이드아웃 — 다음 섹션 배경과 연결됨")
- 수정 시: 기존 txt를 지우지 말고 `G##-용도-v2.txt`로 새로 작성한다. 첨부 세트가 바뀌면 첨부 폴더도 v2로.

### 9-1. 첨부 에셋 세팅 규칙 (이미지 G## / 영상 V## 공통)

생성 모델에 프롬프트와 **함께 입력해야 하는 파일**이 있는 경우(image-to-image 합성, 스타일
레퍼런스, i2v 시드 프레임 등), 에이전트가 프롬프트 작성 시점에 다음까지 세팅한다:

```
assets/prompts/
├── G03-hero-bg.txt                  ← 프롬프트 (첨부 없으면 이 파일 하나로 끝)
└── G03-hero-bg-attachments/         ← 첨부가 있으면 에이전트가 이 폴더를 만들고 파일을 복사해둔다
    ├── ref-01-product.png           ← assets/provided/의 P 에셋에서 복사 (원본은 그대로 둠)
    └── ref-02-mood.jpg
```

- 첨부 후보: P 에셋(제품 누끼·실사 컷), 기존 G 에셋(앞서 생성한 이미지를 시드로 재사용),
  외부 레퍼런스 이미지. **원본을 옮기지 말고 복사**해서 세팅한다 (원본 위치는 매니페스트 기준 유지).
- 파일명은 `ref-##-역할` 형식으로 다시 지어, txt의 [첨부 에셋] 목록과 1:1로 대응시킨다.
- 사람의 실행 절차가 "폴더 열기 → 프롬프트 복사 → 첨부 전부 업로드 → 생성"으로 끝나야 한다.
  첨부의 용도 구분(합성 대상인지, 스타일 참고인지)은 txt에 이미 적혀 있으므로 사람이 고민할 게 없어야 한다.
- 아직 준비되지 않은 P 에셋을 첨부해야 하는 경우: 매니페스트에 의존 관계를 명시하고
  ("G03은 P01 수급 후 첨부 세팅 예정"), P 에셋 도착 시 에이전트가 첨부 폴더를 완성한다.

## 10. Figma 작업 규칙 요약

1. `/figma-use` 스킬 로드 없이 `use_figma`를 호출하지 않는다. 새 파일은 `/figma-create-new-file` 먼저.
2. 파일명은 생성 시에만 지정 가능 → 생성 시 `[Claude]` 마커 포함.
3. 텍스트는 100% 네이티브 텍스트 레이어. 이미지에 텍스트를 굽지 않는다.
4. 레이어 이름을 의미 있게 짓는다 (`H1-메인카피`, `BG-hero`, `IMG-P01-product`).
5. 반복 구조는 오토레이아웃(카드·표는 hug). 색·폰트·크기는 blueprint 토큰만 사용 (임의 값 금지).
6. 프레임 이름은 순번+역할 (`01-훅`, `02-문제제기`, ...).

## 11. 셀프 QA 체크리스트 (공유 전 필수)

> 플레이북의 추가 체크 항목까지 합쳐서 검수한다. 하나라도 실패면 수정 후 재검수.

- [ ] **타이포 스케일 검산**: 본문·캡션이 플레이북 하한 이상인가 (상세페이지: 본문 ≥26px@1000px)
- [ ] **축소 판독 테스트**: maxDimension 400(광고 200) 캡처에서 본문 판독·숫자 강조 확인
- [ ] **셀링포인트 노출**: 핵심 숫자가 각 섹션 최상위 위계로 보이는가, 근거 없는 주장이 없는가
- [ ] **밀도**: 텍스트만 있는 섹션 없음, 빈 세로 구간 ≤ 폭의 30%, 카드 바닥 빈 공간 없음
- [ ] **레퍼런스 대조**: 나란히 놓고 밀도·타이포·증거에서 밀리지 않는가
- [ ] 텍스트 오버플로/잘림 없음, 어절 단위 줄바꿈
- [ ] 배경 위 텍스트 대비 충분 (필요 시 스크림/그라데이션 오버레이 추가)
- [ ] 정렬·간격 일관성 (같은 역할 요소는 같은 위치·크기)
- [ ] 컬러·폰트가 blueprint 토큰과 일치
- [ ] 누끼 가장자리 깨짐·저해상도 업스케일 흔적·생성 배경 스트레치 흔적 없음
- [ ] 프레임 간 톤 통일성 (연작으로 보이는가), 섹션 이음새 자연스러움
- [ ] 로고·인증마크 등 필수 요소 누락 없음, 심의 위험 표현 점검(디스클레이머 포함)
- [ ] (영상) 클립에 글자·워터마크 없음, 씬 연결 시 톤 점프 없음

## 12. 안티패턴 — 하지 말 것

- ❌ 리서치(research.md) 없이 카피를 지어내며 설계 시작 — 근거 없는 카피는 지면 낭비
- ❌ 설계 없이 바로 Figma/Higgsfield부터 만지기
- ❌ **웹 UI 감각의 타이포(본문 16~18px)로 커머스 지면을 조립** — 축소되면 판독 불가
- ❌ 에셋이 다 준비되기 전에 조립 시작 (임시 이미지로 조립하면 결국 재작업)
- ❌ GPT Image에게 텍스트 포함 이미지를 요구 (문구 수정 = 전체 재생성 지옥)
- ❌ 2:3보다 긴 프레임을 생성 이미지 1장으로 채우기 (스트레치·아웃페인팅 포함)
- ❌ 인증·근거를 실물 이미지 없이 텍스트 뱃지로 때우기
- ❌ 프롬프트를 채팅에만 남기고 txt 파일로 안 남기기
- ❌ 이전 작업물·레퍼런스의 레이아웃을 새 작업에 복붙 (과적합 — 측정만 하고 설계는 새로)
- ❌ 에셋을 최소한으로만 요구 (풍부함이 곧 퀄리티)
- ❌ 셀프 QA(축소 판독 포함) 없이 사용자에게 초안 공유
